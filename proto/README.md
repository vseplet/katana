# proto — WebRTC-стриминг экрана macOS в браузер (PoC)

Реализация ТЗ `TZ_webrtc_screenshare_prototype.md`.

Труба: `ffmpeg (avfoundation + VP8/IVF) → Go (pion/webrtc v4) → браузер`.

## Запуск

```bash
cd proto
go run .                 # дефолты: :8080, screen 3, 1280px, 30fps
```

Открыть http://localhost:8080 — должно появиться живое видео рабочего стола.

Флаги: `--addr`, `--screen`, `--width`, `--fps`, `--bitrate`, `--test`.

`--test` — синтетический источник `testsrc2` (lavfi) вместо экрана: отладка
видео-трубы без TCC-разрешения на запись экрана.

## Управление из браузера (на лету)

На странице — панель: **размер картинки**, **частота кадров**, **качество (битрейт)**.
Смена применяется без переподключения: сервер перезапускает ffmpeg с новыми
параметрами, VP8 меняет разрешение на ближайшем кейфрейме. Реализовано
control-сообщением `{type:"config", config:{width,fps,bitrateKbps}}` по тому же WS.

> ⚠️ Размеры выше ~1920p и «нативное» на software-VP8 заметно греют CPU
> (Retina, см. §7 ТЗ). Для высоких разрешений нужен HW-энкод (VideoToolbox) —
> следующий этап.

## Гоча: фриз на первом кадре (исправлено)

avfoundation отдаёт кадры с переменной частотой (VFR), а `WriteSample` использует
фиксированную `Duration`. Без согласования реальная каденция расходится с RTP-часами
и браузер замирает после первого кадра. Лечится форсом CFR на выходе ffmpeg
(`-r <fps> -fps_mode cfr`).

## ⚠️ Разрешение на запись экрана (TCC) — обязательно

`ffmpeg` получает кадры экрана только если **приложение, из-под которого он запущен**, имеет
доступ в `System Settings → Privacy & Security → Screen Recording`.

- Запускать нужно из **реального терминала** (Terminal.app / iTerm / терминал IDE), а не из
  песочницы/агента.
- Без разрешения ffmpeg **молча зависает** после `Overriding selected pixel format` — нет строк
  `Output #0` / `frame=`, в браузере соединение `connected`, но видео чёрное (`videoWidth=0`).
- При первом запуске система должна показать диалог; если нет — добавить терминал вручную и
  перезапустить его.

## Индекс экрана

Плавает между машинами. Проверить перед запуском:

```bash
ffmpeg -f avfoundation -list_devices true -i ""
# ищи "Capture screen 0" — его индекс передать в --screen
```

## Статус проверки

Проверено сквозь DevTools-протокол в Chrome: HTTP-статика, WS-сигналинг, оффер/ответ,
трикл-ICE и `PeerConnection` доходят до состояния `connected`; ffmpeg стартует и его процесс
корректно убивается при закрытии вкладки. Картинка не отрисовалась только из-за отсутствия
TCC-разрешения у процесса, запущенного из-под агента (см. выше) — это внешнее условие, не код.
